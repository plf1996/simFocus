## 1. Executive Summary

### 1.1 Project Overview

simFocus is a greenfield project building an AI-powered virtual focus group platform. The system enables users to create discussion topics, configure AI characters, run multi-character discussions in real-time, and generate insight reports.

### 1.2 Architecture Principles

| Principle | Description | Application to simFocus |
|-----------|-------------|-------------------------|
| **Simplicity First** | Choose the simplest solution that meets requirements | Monorepo with shared types, avoid microservices overhead |
| **Domain-Driven Design** | Structure code around business domains | Separate services: User, Topic, Character, Discussion, Report |
| **Progressive Enhancement** | Build core first, enhance iteratively | MVP with basic features, add intelligence in P1/P2 |
| **Team Adaptability** | Match tech complexity to team capabilities | Clear separation of concerns, well-documented patterns |
| **Cloud-Native Ready** | Containerized from day one | Docker + Docker Compose for local and production |

### 1.3 Key Technical Decisions

| Decision Area | Choice | Rationale |
|--------------|--------|-----------|
| **Repository Structure** | Monorepo | Tight frontend-backend coupling, shared types, single deployment unit |
| **Backend Framework** | FastAPI | Native async, auto OpenAPI, type hints with Pydantic |
| **Frontend Framework** | Vue 3 + TypeScript | Composition API, better TS support than React, smaller bundle |
| **State Management** | Pinia | Simpler than Vuex, excellent TypeScript support |
| **Database ORM** | SQLAlchemy 2.0 | Async support, mature ecosystem, Alembic migrations |
| **WebSocket** | Socket.IO | Auto-reconnection, fallback to polling, Redis pub/sub scaling |
| **API Documentation** | Auto-generated by FastAPI | OpenAPI 3.0, Swagger UI out of the box |

## 2. Project Structure Decision

### 2.1 Monorepo vs Separate Repositories

**Decision: Monorepo**

#### Comparison Analysis

| Criterion | Monorepo | Separate Repos | Winner |
|-----------|----------|----------------|--------|
| **Shared Types** | Single source of truth | Need npm package or copy-paste | Monorepo |
| **Coordinated Changes** | Single commit for both | Multiple PRs, sync issues | Monorepo |
| **Local Development** | One `docker-compose up` | Multiple dev environments | Monorepo |
| **CI/CD Complexity** | Medium | Higher (multi-repo pipelines) | Monorepo |
| **Independent Deployment** | Possible with config | Naturally independent | Separate |
| **Team Size Consideration** | < 10 developers optimal | Better for larger teams | Monorepo |

#### Rationale for Monorepo

1. **Shared Types Critical**: Frontend and backend share extensive types (Character config, API models, DTOs)
2. **Tight Coordination Needed**: Discussion engine requires frontend-backend WebSocket protocol alignment
3. **Single Deployment Unit**: MVP targets single deployment, not microservices
4. **Team Efficiency**: Small team can manage single codebase more effectively
5. **Future Flexibility**: Can extract to separate repos if needed (using tools like nx or turbo)

#### Monorepo Structure Benefits

```
simFocus/
├── backend/          # FastAPI application
├── frontend/         # Vue 3 application
├── shared/           # Shared TypeScript types and utilities
├── docker/           # Docker configurations
├── docs/             # Documentation
├── scripts/          # Development and deployment scripts
└── docker-compose.yml
```

### 2.2 Deployment Strategy

| Environment | Deployment Model |
|-------------|-----------------|
| **Local Development** | Docker Compose (all services) |
| **Staging** | Docker Compose or Kubernetes (single namespace) |
| **Production** | Separate frontend/backend deployment with CDN |

### 4.1 Backend Layer Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                      API Layer (routes)                     │
│  Request validation, authentication, response formatting   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Service Layer (business logic)           │
│  Orchestrates operations, calls repositories and LLM       │
└─────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
┌──────────────────────────┐    ┌──────────────────────────┐
│  Repository Layer        │    │  LLM Provider Layer      │
│  (Data Access)           │    │  (External APIs)         │
└──────────────────────────┘    └──────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Data Layer (PostgreSQL + Redis)          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Service Layer Organization

| Service | Responsibilities | Dependencies |
|---------|-----------------|--------------|
| **AuthService** | Login, register, token generation, password reset | UserRepository, encryption utils |
| **UserService** | Profile management, preferences, stats | UserRepository, cache |
| **TopicService** | Topic CRUD, draft management, search | TopicRepository |

### 9.3 Frontend Composable Pattern

```typescript
// frontend/src/composables/useDiscussion.ts
import { ref, computed } from 'vue';
import { useSocket } from './useSocket';
import { discussionService } from '@/services/discussion.service';
import type { Discussion, DiscussionMessage } from '@/types';

export function useDiscussion(discussionId: string) {
  const discussion = ref<Discussion | null>(null);
  const messages = ref<DiscussionMessage[]>([]);
  const isLoading = ref(false);
  const error = ref<Error | null>(null);

  const socket = useSocket();

  const isRunning = computed(() => discussion.value?.status === 'running');
  const progress = computed(() => discussion.value?.progress_percentage ?? 0);

  async function fetchDiscussion() {
    isLoading.value = true;
    try {
      discussion.value = await discussionService.getById(discussionId);
    } catch (e) {
      error.value = e as Error;
    } finally {
      isLoading.value = false;
    }
  }

  async function startDiscussion() {
    await discussionService.start(discussionId);
    discussion.value = await discussionService.getById(discussionId);
    socket.subscribe(discussionId);
  }

  function pauseDiscussion() {
    socket.emit('control', { control_type: 'pause' });
  }

  // Cleanup
  onUnmounted(() => {
    socket.unsubscribe(discussionId);
  });

  return {
    discussion,
    messages,
    isLoading,
    error,
    isRunning,
    progress,
    fetchDiscussion,
    startDiscussion,
    pauseDiscussion,
  };
}
```
